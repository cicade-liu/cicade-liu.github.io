桥介数物开发环境的基础设施技能，日常开发必备

第一部分：Linux 系统使用

1.1 为什么必须学 Linux？

桥介数物的开发环境完全基于 Linux：

依赖工具 运行环境 说明
ROS1 Ubuntu 20.04 机器人操作系统，仅支持 Linux
Isaac Gym Ubuntu 20.04/22.04 NVIDIA 官方仅支持 Linux
OCS2 Ubuntu 20.04 ETH 框架，Linux 编译
Docker 部署 Linux Server 实机控制板跑 Ubuntu
CUDA Linux GPU 训练，NVIDIA 驱动

结论：不会 Linux = 无法在桥介的开发环境中工作。

补充：Linux 相比 Windows 的核心优势（在机器人开发中）

  • 包管理器：apt install 一键安装依赖，不需要去官网下载安装包、配置环境变量。ROS、CUDA、编译工具链全部通过包管理器统一管理。
  • 命令行优先：所有工具都可以通过命令行驱动，意味着所有操作都可以脚本化、自动化。训练脚本启动 → 自动编译 → 自动运行 → 自动收集日志，不需要手动点击鼠标。
  • 文件权限模型：Linux 的 rwx 权限模型清晰且可编程，SSH 远程登录、多用户协作天然安全。

1.2 Shell 基础命令

原理①：Shell 是什么？命令是怎么被执行的？

Shell 是用户和 Linux 内核之间的命令解释器。你在终端输入 ls -la，Shell 做的事：

核心系统调用：

  • fork()：完整复制当前进程。父进程返回子进程 PID，子进程返回 0——这是 Linux 中创建新进程的唯一方式。
  • exec()：用新程序替换当前进程的整个内存空间。旧程序被「覆盖」，不再存在。
  • wait()：父进程等待子进程结束，回收资源。

类比：Shell 就像一个项目经理。你告诉它「跑 ls」，它不自己跑，而是 fork 出一个新员工（子进程），让这个员工去 exec ls。经理 wait 员工干完活，回收工位，然后等你下一个指令。

原理②：管道 | 的底层实现

Unix 哲学：一个命令只做一件事，但把它做到极致。ls 只列文件，grep 只搜文本，cat 只输出内容。真正的力量来自把它们用管道组合。

原理③：标准流与重定向

每个进程启动时，内核自动打开三个文件描述符：

文件描述符 名称 默认指向 用途
0 stdin 键盘 程序读取输入
1 stdout 屏幕 程序正常输出
2 stderr 屏幕 程序错误输出

重定向的本质：用 dup2() 系统调用改变这些文件描述符的指向。

为什么区分 stdout 和 stderr？ 正常结果和错误信息分开，你就可以 cmd > output.txt 2> error.log，正常结果存入文件，错误单独记录。管道 | 只传递 stdout，不传 stderr——这是设计，方便你一边看错误信息一边处理正常输出。

原理④：一切皆文件（VFS）

Linux 的**虚拟文件系统（VFS）**层让所有资源看起来都像文件。不仅硬盘上的文件和目录是文件：

  • 键盘输入 → /dev/stdin（读这个文件就是读键盘）
  • 屏幕输出 → /dev/stdout（写这个文件就是写屏幕）
  • 进程信息 → /proc/12345/status（读这个文件就能看到进程状态）
  • 网络连接 → socket 也是文件描述符（send()/recv() 就是写/读文件）

这个统一抽象意味着：你可以用同一套 open/read/write/close 操作一切资源。

文件系统与路径

Linux 文件系统是一棵以 / 为根的树。没有 Windows 的 C 盘 / D 盘概念，所有存储设备都挂载到这棵树的某个分支上。

原理⑤：文件系统 —— inode、硬链接与软链接

理解文件系统，首先要区分两个概念：

概念 存储位置 存什么
文件名 目录中 一个「文件名 → inode号」的映射
inode 磁盘上 inode 表中 权限、大小、时间戳、数据块指针

目录的本质：目录是一个特殊的文件，内容是一张「文件名 → inode号」的映射表。ls -i 可以查看每个文件的 inode 号。

类比：inode 是「房产」，硬链接是「多个门牌号指向同一套房」，软链接是「纸条上写了地址」。

文件删除的真正含义：rm 不是擦除磁盘数据，而是：

  1. 删除目录中的文件名→inode记录
  2. inode中的「链接计数」减1
  3. 当链接计数归零且没有进程打开该文件时，inode和数据块被标记为「空闲」
  4. 这就是为什么 rm 很快（不实际擦数据），以及被删文件有时能恢复

原理⑥：Linux 目录结构（FHS 标准）

Linux 遵循 FHS（Filesystem Hierarchy Standard），每个目录有固定用途：

目录 全称 用途 桥介开发中会遇到什么
/bin Binaries 基本命令（ls, cp, cat） 日常使用
/sbin System Binaries 系统管理命令（需要 root） ifconfig 等
/usr Unix System Resources 用户程序和数据 /usr/local/cuda（CUDA安装位置）
/etc Editable Text Config 配置文件 /etc/apt/sources.list（软件源）
/var Variable 可变数据（日志、缓存） /var/log/syslog
/tmp Temporary 临时文件（重启清空） 编译中间文件
/home Home 用户主目录 ~/catkin_ws（你的工作空间）
/opt Optional 第三方软件 /opt/ros/noetic（ROS安装目录）
/proc Process 内核信息（伪文件系统） /proc/cpuinfo、/proc/meminfo
/dev Devices 设备文件 /dev/ttyUSB0（串口）、/dev/nvidia0（GPU）
/mnt Mount 临时挂载点 挂载外接硬盘

权限管理原理

每个文件有三组权限：拥有者（u）、所属组（g）、其他人（o）。每组有三种操作：读（r=4）、写（w=2）、执行（x=1）。

为什么要学权限？ 编译出的可执行文件没有 x 权限就无法运行，SSH 私钥权限太宽（如 644）会被 SSH 拒绝连接。机器人上误改关键文件权限可能导致系统崩溃。

进程管理原理

进程是什么？ Linux 中每个运行的程序是一个进程，有一个唯一的 PID（Process ID）。内核分配独立的虚拟内存空间给每个进程，进程之间互不干扰——一个进程崩溃不会影响其他进程（和 Windows 98 的蓝屏死机说再见）。

进程的创建：Linux 中创建新进程的唯一方式是 fork()。fork() 完整复制当前进程的内存空间（写时复制优化），只有返回值不同：

  • 父进程中 fork() 返回子进程的 PID
  • 子进程中 fork() 返回 0

Shell 执行命令就是 fork() + exec() 的组合：先 fork 自己，子进程再 exec 目标程序。

进程状态：

状态 符号 含义 典型场景
Running R 正在运行或在运行队列中等待 CPU 正常程序
Sleeping S 等待事件（如网络数据、用户输入） sleep 10
Uninterruptible Sleep D 等待 I/O，不可被信号打断 等待磁盘读写
Stopped T 被 SIGSTOP 暂停，等待 SIGCONT 恢复 调试器断点
Zombie Z 子进程已退出但父进程未 wait 父进程有 bug，没回收子进程

信号机制：信号是 Linux 的异步通知机制。不是轮询，而是内核直接打断进程。

为什么优先用 kill（不带 -9）而不要上来就 kill -9？ kill -9 不给进程任何清理机会——RL 训练中直接 kill -9 可能丢失未保存的 checkpoint、损坏日志文件。好的实践是：先 kill（SIGTERM），等几秒，如果进程还在，再 kill -9。

孤儿进程与僵尸进程：

  • 父进程死了，子进程被 init（PID=1）收养，成为孤儿进程——正常，无害
  • 子进程死了但父进程没 wait()，子进程残留在进程表中，成为僵尸进程——占用 PID，积累多了无法创建新进程。top 中看到 Z 状态就是僵尸

为什么要学进程管理？ 训练脚本卡死时需要 kill；GPU 被占时需要看 nvidia-smi 确认是谁在用；机器人控制进程 CPU 跑满时需要 top/htop 定位问题。

1.3 ROS 开发环境

原理：ROS 是什么？为什么需要它？

ROS（Robot Operating System）不是真正的操作系统，而是一个分布式通信框架。它解决的核心问题：机器人上的不同程序（节点）如何相互通信？

  • 节点（Node）：一个独立的程序，如「摄像头驱动」「SLAM 建图」「路径规划」
  • 话题（Topic）：节点间的通信管道，发布者（Publisher）往话题上发数据，订阅者（Subscriber）从话题上收数据
  • 主节点（roscore）：相当于电话总机，负责帮节点互相找到对方

类比：ROS 就像公司内部的 Slack 频道。每个节点是一个人，话题是频道。有人往 #joint_states 频道发关节角度，有人订阅这个频道来收数据、做控制。

1.5 环境配置

原理：PATH 和环境变量

当你在终端输入 python3，Shell 怎么知道这个程序在哪里？答案是 PATH 环境变量。它是一个目录列表，Shell 按顺序在这些目录里找可执行文件。

同理，ROS 用 ROS_PACKAGE_PATH 找包，CUDA 用 CUDA_HOME 找 CUDA 安装位置。环境变量本质是一个「全局字典」，任何程序都能读取。

为什么需要虚拟环境？ 不同项目可能依赖同一个库的不同版本。项目 A 需要 PyTorch 1.13，项目 B 需要 PyTorch 2.0。全局安装会冲突。虚拟环境给每个项目独立的包空间，互不干扰。

第二部分：Git 版本管理

2.1 Git 的核心原理

在学命令之前，先理解 Git 的底层模型，否则你会永远停留在「死记硬背命令」的阶段。

Git 的四种对象：它是怎么存储你的代码的？

Git 不是存「文件之间的差异（diff）」，而是存完整快照（snapshot）。它的数据库只有四种对象类型：

对象 存什么 类比
blob 文件内容（不含文件名） 一张写了内容的纸
tree 目录结构 =「文件名 + blob/tree 的哈希」列表 一个文件夹，装着纸和子文件夹
commit 指向一个 tree，包含作者、时间、提交信息、parent commit(s) 一个带签名的信封，封装了文件夹
tag 指向某个 commit 的固定引用（用于标记版本号如 v1.0） 贴在信封上的标签

所有对象都以 SHA-1 哈希值作为唯一标识（如 a3f1b2c...）。这是内容寻址：相同内容 → 相同哈希 → Git 只存一份。这就是为什么 Git 能高效存储——你复制一份大文件到三个目录，Git 只存一个 blob，三个 tree 各自指向它。

.git 目录里面到底有什么？

有了这个结构，就能理解 Git 的核心操作：

  • git add → 计算文件 SHA-1，在 objects/ 存 blob，更新 index
  • git commit → 根据 index 创建 tree 对象，再创建 commit 对象
  • git checkout feat/xxx → 修改 HEAD，把目标 commit 的 tree 展开到工作区

Git 是有向无环图（DAG）

Git 的每一次 commit 不是保存整个项目的压缩包，而是保存一个指向「当前项目状态」的快照对象。这些 commit 对象通过父子关系连起来，形成一张有向无环图。

  • 每个字母是一个 commit
  • main 和 feat/new_controller 是分支（指向某个 commit 的指针）
  • 合并（merge）就是在这个图上新建一个有两个父节点的 commit

三个区域：工作区 → 暂存区 → 仓库

区域 通俗理解 Git 命令
工作区（Working Directory） 你正在编辑的文件 你直接用编辑器改
暂存区（Staging Area / Index） 「这次要提交的」文件列表 git add
仓库（Repository / HEAD） 已提交的历史记录 git commit

为什么要有暂存区？ 它让你可以选择性地提交。你改了 5 个文件，但这次只想提交其中 2 个。git add 这两个，git commit 时就只会提交它们。没有暂存区的版本控制系统（如 SVN）做不到这点。

场景 Git 操作

2.3 基础工作流

2.4 常见场景处理

reset 三种模式的区别：

  • --soft：只撤销 commit，修改保留在暂存区（方便重新 commit）
  • --mixed（默认）：撤销 commit 和暂存，修改保留在工作区
  • --hard：全部丢弃，修改也不保留（不可逆，慎用）

rebase vs merge：

  • merge：保留真实的分支历史，产生一个合并 commit。历史图有分叉。
  • rebase：把当前分支的 commit「搬到」目标分支的最新 commit 之后，历史变成一条直线。更整洁，但会改写历史 —— 永远不要对已 push 的分支做 rebase。

2.5 Git 最佳实践（桥介风格）

  1. Commit Message 规范：

    遵循 Conventional Commits 规范：type: 简短描述。type 明确告诉 reviewer 这个 commit 做了什么类型的改动。

  2. 分支命名：

    • feat/xxx：新功能（feature）
    • fix/xxx：修 bug
    • exp/xxx：实验性代码（experiment，别合入 main）

  3. 每次 commit 只做一件事：不要一个 commit 改了控制器又改了 URDF 又删了文档。出问题时 git bisect 才能精确定位。

  4. push 前先 pull：git pull --rebase 保持线性历史，避免无意义的 merge commit。

第三部分：Docker 基础（进阶）

3.1 原理：Docker 解决什么问题？

团队中每个人的电脑环境不一样（Ubuntu 版本、CUDA 版本、ROS 版本），A 能编译的代码 B 编译报错。这就是经典的「在我电脑上能跑」问题。

Docker 的解决方案：把代码和它依赖的整个操作系统环境打包成一个镜像（Image）。任何人拿到这个镜像，就能在一个隔离的**容器（Container）**中运行，环境完全一致。

镜像 vs 容器：镜像是「菜谱」（静态的模板），容器是「做出来的菜」（运行中的实例）。一个镜像可以启动多个容器。

为什么桥介需要 Docker？ 跨机器人部署时，每台机器人的 Ubuntu 版本、驱动版本、依赖库版本都可能不同。Docker 保证训练好的策略在任何机器人的控制板上都能一模一样地运行。
